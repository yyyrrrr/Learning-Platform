# 团队分工

## 说明

- 保持现有 11 个模块划分不变，避免大幅调整已达成的一致意见。
- 本文档在原有“功能模块分工”基础上，补充“责任边界、依赖关系、交付物、目录归属、协作规则”，用于降低 11 人并行开发时的阻塞和文件冲突风险。
- 每位成员仍需至少独立负责一个模块/子模块，并共同参与知识图谱构建；但“共同参与”不等于“共同改所有文件”，知识图谱相关公共文件仍需有明确 owner。

## 协作原则

### 1. 保留模块主责，但补充共享底座 owner

11 人并行开发时，最容易冲突的不是业务页面，而是公共层。为避免“所有人都能改，最后谁都说不清”，以下 3 个模块除业务职责外，还承担共享底座 owner 角色：

- **项目仓库 + 技术架构 + 审核 PR**：
  - 负责仓库结构、分支规范、接口规范、公共配置、后端基础骨架、主分支合并规则
- **项目文档 + 全局协调**：
  - 负责前端基础骨架、文档模板、进度同步、测试记录、会议纪要、风险追踪
- **知识图谱构建（数据层）**：
  - 负责知识点字段定义、关系类型、命名规范、导入格式、图谱数据版本

其他 8 个模块默认是业务功能 owner，不直接拥有公共契约的最终决定权。

### 2. 主责模块 != 独占整个链路

每个模块负责人对“自己模块最终能否交付”负责，但不意味着要独自完成所有前端、后端、数据工作。实际执行时采用：

- **主负责人**：对模块整体进度、接口对接、验收结果负责
- **协作方**：按约定提供接口、数据或组件，不越权修改对方主目录

### 3. 先冻结契约，再并行开发

在核心开发开始前，至少需要先冻结以下 4 份契约：

- 知识点数据模型契约
- Neo4j 关系类型与推荐路径相关契约
- 前后端 API 契约
- 前端公共层契约（路由、状态管理、API 封装、公共组件）

若这 4 项未冻结，所有模块都可能因共享文件和接口反复修改而阻塞。

## 公共契约与 owner

| 契约/公共层 | 主 owner | 协作方 | 说明 |
|------------|---------|-------|------|

| 接口规范、统一响应、错误码、鉴权方案 | 项目仓库 + 技术架构 + 审核 PR | 各后端模块负责人 | 防止各模块各写一套接口风格 |
| 文档模板、进度记录、测试记录 | 项目文档 + 全局协调 | 全员 | 统一文档粒度和命名 |
| 知识点字段、命名规范、关系类型、导入格式 | 知识图谱构建（数据层） | 学习目录、统计分析、可视化、知识点讲解 | 图谱相关模块必须遵循同一数据口径 |
| 前端路由、公共组件、全局 store、API 基类 | 项目文档 + 全局协调 | 所有前端模块 | 前端公共层集中维护，业务模块按规范接入 |
| 集成测试清单、联调时间表、发布前检查 | 项目文档 + 全局协调；视频讲解 + 全局整合 | 全员 | 一个偏流程，一个偏落地联调 |

## 模块边界说明

### 1. 项目仓库 + 技术架构 + 审核 PR

- **主责边界**：
  - 搭建并维护后端项目基础骨架
  - 定义分支、PR、目录、接口、异常处理等公共规范
  - 审核公共层改动，控制主分支质量
- **上游依赖**：
  - 无
- **对外交付**：
  - 后端基础骨架
  - 公共代码规范
  - API 规范文档
  - PR 合并规则
- **不负责**：
  - 不替各业务模块实现具体业务页面和核心业务逻辑

### 2. 项目文档 + 全局协调

- **主责边界**：
  - 搭建并维护前端项目基础骨架
  - 维护需求、架构、接口、测试、会议纪要等文档
  - 跟踪模块进度，记录风险和阻塞项
  - 组织联调和验收材料
- **上游依赖**：
  - 各模块提供真实进度和变更信息
- **对外交付**：
  - 前端基础骨架
  - 统一文档目录
  - 进度表、风险表、测试记录
  - 周会纪要和验收材料
- **不负责**：
  - 不代替各模块负责人做技术设计决策

### 3. 知识图谱构建（数据层）

- **主责边界**：
  - 维护知识点清单、属性字段、层级关系、关联关系
  - 维护 Neo4j 导入格式、关系类型、命名规范
  - 管理图谱数据版本，避免多人直接改同一份图谱源数据
- **上游依赖**：
  - 全体成员共同参与知识点梳理
- **对外交付**：
  - 知识点数据字典
  - 图谱源数据文件
  - Neo4j 导入脚本或导入说明
  - MySQL 与 Neo4j 字段映射说明
- **不负责**：
  - 不直接实现图谱可视化页面和统计分析页面

### 4. 知识图谱交互式可视化

- **主责边界**：
  - 实现图谱展示、缩放、平移、搜索、高亮、节点点击
  - 根据学习状态展示节点样式变化
- **上游依赖**：
  - 知识图谱构建模块提供稳定的节点/边数据结构
  - 学习跟踪模块提供学习状态数据
- **对外交付**：
  - 图谱页面前端
  - 图谱渲染组件
  - 图谱交互所需接口定义
- **不负责**：
  - 不定义知识点字段和关系命名

### 5. 学习目录功能

- **主责边界**：
  - 实现基于图谱层级的目录树
  - 展示学习状态
  - 调用推荐学习路径相关能力并完成前端交互
- **上游依赖**：
  - 知识图谱构建模块提供层级关系
  - 学习跟踪模块提供学习状态
  - 推荐路径能力完成后端接口
- **对外交付**：
  - 学习目录页面
  - 目录树接口需求说明
  - 推荐路径展示交互
- **不负责**：
  - 不单独决定推荐算法规则

### 6. 知识点详细讲解（图文）

- **主责边界**：
  - 实现知识点详情页模板
  - 组织定义、图示、操作讲解、复杂度分析等内容呈现
  - 对接代码示例、视频、练习题入口
- **上游依赖**：
  - 知识图谱构建模块提供知识点基础信息
  - 代码实现与在线运行模块提供代码展示能力
  - 视频模块和习题模块提供关联内容
- **对外交付**：
  - 知识点详情页
  - 内容模板规范
  - 详情页所需数据字段清单
- **不负责**：
  - 不负责在线运行引擎本身

### 7. 代码实现与在线运行

- **主责边界**：
  - 实现代码编辑、语法高亮、运行结果展示
  - 负责运行接口、错误信息返回、可视化步骤序列协议
  - 与可视化组件联动
- **上游依赖**：
  - 架构 owner 先确定运行环境和安全策略
  - 知识点讲解模块提供代码示例接入场景
- **对外交付**：
  - 代码编辑器页面/组件
  - 运行接口
  - 步骤序列协议文档
- **不负责**：
  - 不负责全部知识点内容编写

### 8. 习题模块

- **主责边界**：
  - 负责题库录入、答题交互、判分、错题记录、答案解析
  - 维护题目字段和做题流程
- **上游依赖**：
  - 知识点讲解模块提供知识点与题目挂接入口
  - 学习跟踪模块消费做题结果
- **对外交付**：
  - 习题页面
  - 题库管理接口
  - 判分与记录接口
- **不负责**：
  - 不负责用户总体学习统计口径

### 9. 学习跟踪与统计分析

- **主责边界**：
  - 记录学习行为、进度、做题情况、视频进度
  - 输出统计图表、学习报告、薄弱点分析
  - 负责与知识图谱关联的分析口径
- **上游依赖**：
  - 目录、图谱、习题、视频等模块提供行为事件
  - 知识图谱构建模块提供关系数据
- **对外交付**：
  - 行为记录接口
  - 统计分析接口
  - 报告页面和导出方案
- **不负责**：
  - 不直接维护知识图谱源数据

### 10. AI 辅助功能

- **主责边界**：
  - 对接 AI 问答、代码分析、代码生成能力
  - 维护多轮对话的接口和上下文逻辑
- **上游依赖**：
  - 架构 owner 先确定模型选型、密钥管理和调用方式
  - 代码实现与在线运行模块提供代码输入场景
  - 知识点讲解和习题模块提供上下文来源
- **对外交付**：
  - AI 对话接口
  - 代码分析接口
  - 代码生成接口
- **不负责**：
  - 不负责全站所有输入框的前端实现

### 11. 视频讲解 + 全局整合

- **主责边界**：
  - 负责视频播放、观看进度记录
  - 负责联调排期、整体验收、展示前问题收敛
  - 负责 UI 一致性检查和跨模块体验检查
- **上游依赖**：
  - 其他模块按约定提供可联调版本
- **对外交付**：
  - 视频模块
  - 联调清单
  - 提测版本和问题单
- **不负责**：
  - 不作为所有模块公共代码的唯一实现者
  - 不替代各模块 owner 修复其本模块核心问题

## 目录归属建议

为减少文件冲突，建议按“主目录 owner + 公共目录审批”执行。

### 一、公共目录

以下目录原则上不能被所有人随意修改，需由对应 owner 审核：

| 目录/文件 | 主 owner | 说明 |
|----------|---------|------|
| `docs/` | 项目文档 + 全局协调 | 文档统一入口，但各模块可维护自己的子文档 |
| `data-layer/scripts/mysql/init.sql` | 知识图谱构建（数据层） + 项目仓库 + 技术架构 + 审核 PR | 涉及全局数据模型，变更必须同步 |
| `data-layer/scripts/mysql/mysql-modules/` | 各模块负责人 | 各模块数据定义 |
| `src/main/resources/application*.yml` | 项目仓库 + 技术架构 + 审核 PR | 配置变更影响全局环境 |
| `src/main/java/com/learningplatform/common/` | 项目仓库 + 技术架构 + 审核 PR | 统一响应、异常、工具类等公共层 |
| `src/main/java/com/learningplatform/config/` | 项目仓库 + 技术架构 + 审核 PR | 公共配置由架构 owner 统筹 |
| `frontend/src/router/` | 项目文档 + 全局协调 | 路由是公共层 |
| `frontend/src/api/` | 项目文档 + 全局协调 | API 基类和请求封装统一管理 |
| `frontend/src/stores/` | 项目文档 + 全局协调 | 全局状态结构避免多人随意扩展 |
| `frontend/src/components/` 下公共组件目录 | 项目文档 + 全局协调 | 公共组件命名和抽象需统一 |

### 二、业务目录

以下目录按模块主负责人主导开发：

| 模块 | 建议主目录 |
|------|-----------|
| 知识图谱交互式可视化 | `frontend/src/components/KnowledgeGraph/`、`frontend/src/views/Graph/` |
| 学习目录功能 | `frontend/src/views/LearningPath/` |
| 知识点详细讲解（图文） | `frontend/src/views/KnowledgeDetail/` |
| 代码实现与在线运行 | `frontend/src/components/CodeEditor/`、`frontend/src/components/Visualizer/`、`frontend/src/views/CodePlayground/` |
| 习题模块 | `frontend/src/views/Exercise/`、对应后端习题相关包 |
| 学习跟踪与统计分析 | `frontend/src/views/Statistics/`、对应后端统计相关包 |
| AI 辅助功能 | `frontend/src/views/AIChat/`、对应后端 AI 相关包 |
| 视频讲解 + 全局整合 | `frontend/src/components/Video/`、对应后端视频相关包 |

说明：

- 若当前仓库尚未创建这些目录，可在基础框架搭建完成后统一建立。
- 涉及跨模块公共抽象时，应先提交到公共层 owner 评审，不建议业务模块直接把自己的实现塞进公共目录。

## 接口与联调规则

### 1. 接口先文档化

任何模块在开发接口前，至少先写清楚：

- 接口路径
- 请求参数
- 响应结构
- 错误码
- 是否需要登录
- 给谁使用

未成文档的接口，不视为稳定接口。

### 2. 模块之间通过契约协作，不通过口头假设协作

例如：

- 学习目录模块不能假设图谱返回字段名，应以数据字典为准
- 统计分析模块不能自行定义“掌握度”字段语义，应先和图谱/进度口径对齐
- AI 模块不能直接读取别的模块私有表结构，应通过服务接口或明确字段契约接入

### 3. 公共文件改动需要额外说明

凡涉及以下内容的 PR，必须在描述中注明“影响模块”：

- 数据库表结构
- Neo4j 节点/边结构
- 公共配置
- 路由结构
- 全局 store
- API 响应结构

## 开发节奏建议

### 第一阶段：冻结底座

- 完成知识点清单与图谱初稿
- 完成数据字典、关系类型、命名规范
- 完成接口规范、目录规范、分支规范
- 搭建基础后端和前端骨架

### 第二阶段：模块并行开发

- 图谱可视化、学习目录、知识点详情、习题模块优先推进
- 在线运行、统计分析、AI 功能同步做接口和方案验证
- 视频模块可后置，但播放器接入方式需提前确定

### 第三阶段：联调与收口

- 按“数据层 → 后端接口 → 前端页面 → 跨模块联动”顺序联调
- 视频讲解 + 全局整合模块负责组织联调，但问题仍由各模块 owner 各自修复

---

## 验收口径

每个模块验收时，除功能是否可用外，还需回答 4 个问题：

- 改了哪些目录和文件
- 依赖哪些上游模块
- 对外提供了哪些接口/组件/数据
- 若模块未完成，当前阻塞点是什么

只有能说明清楚这 4 点，才算真正完成了“一个人负责一个模块”，而不是只负责了一个页面或一段代码。
