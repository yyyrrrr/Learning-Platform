# 数据库交互契约

## 1. 设计原则

1. **Fail fast**：数据库约束（NOT NULL、UNIQUE、外键）是最后一道防线，业务层不依赖数据库静默截断或默认值兜底。
2. **信任内部调用方**：仅在系统边界（Controller 接收的参数、外部 API 返回）做校验；Service 内部调用 Mapper 时，不重复做防御性空判断。
3. **改动最小化**：本契约优先对齐**现有代码与现有表结构**，对历史遗留的不一致予以标注，后续逐步收敛，不一次性推翻重写。

## 2. MySQL 交互契约

### 2.1 命名与映射规范

| 层级 | 规范 | 说明 |
|------|------|------|
| 数据库字段 | 下划线命名（snake_case） | `knowledge_point_id`、`is_deleted` |
| Java 实体字段 | 驼峰命名（camelCase） | `knowledgePointId`、`isDeleted` |
| 映射方式 | MyBatis Plus 自动驼峰转换 | `application.yml` 已配置 `map-underscore-to-camel-case: true`，**禁止**在 XML/注解中额外写 `resultMap` 做非标准映射 |
| 表名 | 全小写，下划线分隔 | `user_exercise_record` |
| 实体类名 | 大写驼峰，与表名对应 | `UserExerciseRecord` |

### 2.2 逻辑删除统一约定

- **全局配置**：`application.yml` 已指定 `logic-delete-field: isDeleted`。
- **基类统一**：所有业务实体必须继承 `BaseEntity`，`isDeleted` 字段定义在基类中，类型为 `Integer`。
- **数据库侧**：各表统一使用 `is_deleted TINYINT NOT NULL DEFAULT 0`。
- **值语义**：`0` = 未删除，`1` = 已删除。
- **禁止项**：禁止各表自行设计其他删除标记字段（如 `deleted`、`status` 兼做删除）；禁止在 Java 代码中手动 set `isDeleted`。

> **历史遗留不一致**：`10_exercise.sql` 中 `exercise.is_deleted` 类型为 `int`（非 `TINYINT`），与 `00_schema.sql` 中其他表的 `TINYINT` 不统一。后续统一收敛为 `TINYINT`。

### 2.3 时间字段统一约定

- **强制字段**：每张业务表必须有 `create_time`、`update_time`。
- **填充方式**：由 `MyMetaObjectHandler` 自动填充，`insert` 时填充 `createTime` 和 `updateTime`，`update` 时填充 `updateTime`。
- **类型对应**：
  - MySQL：`DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP` / `ON UPDATE CURRENT_TIMESTAMP`
  - Java：`LocalDateTime`
- **禁止项**：业务代码禁止手动 `setCreateTime` / `setUpdateTime`。

### 2.4 Entity 与 Mapper 规范

- **继承要求**：所有 Entity 必须继承 `BaseEntity`。
- **主键策略**：使用 `@TableId(type = IdType.AUTO)`，对应 MySQL `AUTO_INCREMENT`。
- **表名注解**：使用 `@TableName("actual_table_name")` 显式声明表名。
- **保留字转义**：字段名若为 MySQL 保留字（如 `options`），使用 `` @TableField("`options`") `` 转义。
- **Mapper 注册**：每个 Mapper 接口独立标注 `@Mapper`；**不启用**全局 `@MapperScan`。
- **Service 继承**：Service 层推荐继承 `ServiceImpl<Mapper, Entity>`，以获得 MyBatis Plus 基础 CRUD 能力。

### 2.5 JSON 字段约定

部分表使用 MySQL JSON 类型或 TEXT 存储 JSON 字符串，Java 侧统一按 `String` 处理，由 Jackson/Fastjson2 做序列化/反序列化。

| 表名 | 字段 | Java 类型 | 内部结构约定 |
|------|------|-----------|-------------|
| `exercise` | `options` | `String` | `["A. xxx", "B. xxx"]` |
| `exercise` | `test_cases` | `String` | `[{"input":"...","expectedOutput":"..."}]` |
| `code_submission` | `analysis_result` | `String` | 由 AI 服务返回的 JSON，结构待定 |

**规则**：
1. 写入前必须在 Service 层完成对象 → JSON 字符串的转换，禁止直接把非 JSON 字符串写入。
2. 读取后若需强类型使用，应在 Service 层用 `ObjectMapper` 反序列化为 DTO，禁止在 Entity 中直接存 `Object`/`Map`。

### 2.6 跨语言共享表约定（Java ↔ Python）

`ai_chat` 表由 **Python AI 服务**（`src/python/AIchat/`）直接读写，Java 后端目前未对接。若未来 Java 侧需要查询或管理该表，必须遵守以下契约：

| 项目 | 约定 |
|------|------|
| Schema 变更权 | 变更 `ai_chat` 表结构时，**Java 侧拥有优先提议权**，Python 侧必须兼容；禁止单方面增减字段 |
| 字段类型 | `user_id` BIGINT, `session_id` VARCHAR(100), `role` VARCHAR(20), `content` TEXT, `create_time` DATETIME |
| `role` 枚举 | `USER`, `ASSISTANT`；Python 侧已使用枚举类 `ChatRole`，Java 侧若对接须保持一致 |
| 索引 | 已有 `idx_user_id`, `idx_session_id`, `idx_create_time`；新增索引须同步通知 Python 侧 |
| 写入方式 | Python 侧使用 `pymysql` 裸 SQL；Java 侧若对接应补全 Entity + Mapper，禁止直接写 JDBC Template 绕过基类约定 |

## 3. Neo4j 交互契约

### 3.1 节点与属性白名单

- **唯一节点标签**：`:KnowledgePoint`
- **必须属性**（创建节点时不可为空）：
  - `neo4j_id`：String，全局唯一，格式 `KP_{模块前缀}_{序号}`，如 `KP_TREE_001`
  - `name`：String，知识点名称
  - `code`：String，知识点编码，与 `knowledge_point.code` 一致
  - `module`：String，模块名；合法值：`SET`, `LINEAR_LIST`, `TREE`, `GRAPH`, `SEARCH`, `SORT`
  - `level`：Integer，层级 `1/2/3`
  - `difficulty`：String，`EASY`, `MEDIUM`, `HARD`
- **可选属性**：`description`, `core_points`, `mysql_id`（用于反向关联 MySQL 记录）

### 3.2 关系类型枚举

| 关系类型 | 方向 | 语义 | 允许的属性 |
|----------|------|------|-----------|
| `PARENT_OF` | 父 → 子 | 层级包含关系 | `order`: Integer，同级排序 |
| `PREREQUISITE` | 前置 → 后置 | 学习依赖（需先学前置） | `reason`: String，说明依赖原因 |
| `RELATED_TO` | 双向或无向 | 弱关联（如栈与递归） | `reason`: String |

**禁止**：未经评审擅自新增关系类型。

### 3.3 查询与写入规范

- **查询必须带标签**：所有 `MATCH` 必须指定节点标签，如 `MATCH (n:KnowledgePoint)`；禁止 `MATCH (n)` 全图扫描。
- **参数化查询**：Cypher 中所有外部传入的字符串必须使用参数（`$param`），禁止字符串拼接。
- **写入权限**：目前 Java 后端**仅有只读查询接口**（`GraphService`）。若后续需要动态维护图谱，必须新增专门的写入 Service，并在事务中保证 MySQL `knowledge_point` 与 Neo4j 节点的一致性。

### 3.4 双数据库一致性约定

- **关联字段**：MySQL `knowledge_point.neo4j_id` 与 Neo4j 节点 `neo4j_id` 属性是一一映射键。
- **当前策略**：**人工维护 + 启动导入**（Cypher 脚本由数据层负责人维护，MySQL 数据由各模块 SQL 维护）。
- **校验方式**：定期执行以下语句比对数量与关键字段：
  ```sql
  -- MySQL
  SELECT code, neo4j_id, name FROM knowledge_point WHERE is_deleted = 0;
  ```
  ```cypher
  -- Neo4j
  MATCH (kp:KnowledgePoint)
  RETURN kp.neo4j_id, kp.code, kp.name
  ORDER BY kp.neo4j_id;
  ```
- **未来演进**：若需运行时同步，优先采用 **Java 侧事务双写**（同一 Service 方法内先写 MySQL，再写 Neo4j），而非定时任务，以保证实时性。

## 5. 新增表/字段的变更流程

1. 模块负责人在 `data-layer/mysql/` 下按命名规范 `NN_模块名.sql` 编写 DDL/DML。
2. 若涉及跨模块共享字段（如 `user_id`、`knowledge_point_id`），必须在 PR 中引用本契约的相关条款。
3. 若涉及 Neo4j 节点/关系变更，同步修改 `neo4j/schema.cql` 或 `neo4j/import/` 下的 Cypher 文件。
4. Java 侧补全 Entity → Mapper → Service → Controller，禁止只有表没有代码的“裸表”。
5. 若 Python 侧也需要感知该表，必须在 `数据库交互契约.md` 的“跨语言共享表”章节登记。
