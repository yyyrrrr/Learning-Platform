## 二、后端

### 🔴 严重

#### 2.1 全局逻辑删除配置与实体脱节
- **位置**：`application.yml:37` / `entity/BaseEntity.java:21` / `entity/Exercise.java:48-49`
- **问题**：
  - `application.yml` 配置了 `logic-delete-field: isDeleted`。
  - `BaseEntity` 中的 `isDeleted` 被注释掉，导致子类无法继承该字段。
  - `Exercise` 自行用 `@TableLogic` 标注了 `deleted`（字段名都不一致）。
  - `User`、`KnowledgePoint`、`LearningProgress` 等实体无逻辑删除字段，但对应 DB 表都有 `is_deleted`。
- **影响**：全局逻辑删除配置名存实亡；新增开发者容易误以为所有实体都已自动支持逻辑删除。
- **建议**：在 `BaseEntity` 中恢复 `isDeleted` 字段并标注 `@TableLogic`，删除各子类的重复定义。

#### 2.2 `ExerciseService` 编程题判题为空壳实现
- **位置**：`service/ExerciseService.java:266-286`
- **问题**：`runTestCases()` 并未真正运行代码，而是检查字符串中是否包含 `"error"` 或 `"Error"`。
- **影响**：编程题提交后永远返回伪结果。
- **建议**：接入 `CodeExecutionService` 的真实执行能力，或至少调用本地 Piston 接口。

---

### 🟠 重要

#### 2.3 响应封装类混用
- **位置**：各 Controller
- **问题**：
  - `UserController`、`GraphController`、`KnowledgePointController`、`CodeController` 使用 `Result<T>`。
  - `ExerciseController` 使用 `ApiResponse<T>`。
- **影响**：前端需要同时兼容两套响应格式。
- **建议**：全项目统一到 `Result<T>`，逐步替换 `ApiResponse<T>`。

#### 2.4 认证硬编码
- **位置**：`controller/ExerciseController.java:82-86`
- **问题**：`getCurrentUserId()` 固定返回 `1L`。
- **影响**：所有答题记录、错题本都挂在同一用户上。
- **建议**：接入 Spring Security 或 JWT，至少先实现登录态传参。

#### 2.5 `KnowledgePointController` 直接暴露实体
- **位置**：`controller/KnowledgePointController.java:30-33`
- **问题**：`save(@RequestBody KnowledgePoint point)` 直接接收实体，无 DTO 隔离和字段校验。
- **影响**：可能把非法字段（如 `id`）写入数据库。
- **建议**：定义 `CreateKnowledgePointRequest` DTO，配合 `@Valid` 校验。

#### 2.6 全局异常处理缺失
- **位置**：全项目
- **问题**：已定义 `BusinessException`，但无 `@ControllerAdvice` / `@ExceptionHandler`。
- **影响**：业务异常抛出后，前端收到 Spring 默认的 500 HTML 页面，而非 JSON。
- **建议**：新增全局异常处理类，统一封装异常响应。

### 🟡 一般

#### 2.7 依赖冗余
- **位置**：`pom.xml:78-81`
- **问题**：引入了 `fastjson2`，但全项目 Java 源码中零引用。Spring Boot 默认使用 Jackson。
- **建议**：移除冗余依赖，或明确在配置中启用 Fastjson2 作为默认 JSON 处理器。

#### 2.8 MyBatis XML 配置指向空目录
- **位置**：`application.yml:28`
- **问题**：`mapper-locations: classpath*:/mapper/**/*.xml`，但 `src/main/resources/mapper/` 不存在。
- **建议**：删除该配置，或补充需要的 XML Mapper 文件。

#### 2.9 CORS 双重配置且其中一个违规
- **位置**：`config/CorsConfig.java` / `config/WebConfig.java`
- **问题**：两处都配置了 CORS。`CorsConfig` 使用 `allowedOriginPatterns("*")` + `allowCredentials(true)`，浏览器规范禁止两者同时使用，会被浏览器拒绝。
- **建议**：保留 `WebConfig`（限定 localhost:5173），删除 `CorsConfig`。

#### 2.10 `ExerciseService` 手动维护时间戳
- **位置**：`service/ExerciseService.java:118, 299, 313`
- **问题**：多处手动设置 `createTime` / `updateTime`，但 `MyMetaObjectHandler` 已配置自动填充。
- **建议**：删除手动赋值，信任自动填充机制。

#### 2.11 密码明文存储
- **位置**：`controller/UserController.java:22-26`
- **问题**：`save()` 直接透存 `user.getPassword()`，无哈希。
- **建议**：使用 BCrypt 等算法加密后存储。

#### 2.12 实体类继承不一致
- **位置**：`entity/UserExerciseRecord.java`
- **问题**：不继承 `BaseEntity`，自己声明 `createTime` 但缺少 `updateTime`，与其他实体模式不一致。
- **建议**：统一继承 `BaseEntity`。

## 四、后端修复方案（2026-05-07 补充）

> 范围：仅后端 Java 代码与数据库脚本。前端不在本仓库，响应格式统一等改动需知会前端同学配合。

---

### P0 严重（导致运行时错误，优先修复）

#### 1. 统一 `exercise` 表结构（对应 1.1）

**目标**：以 `exercise.sql` 的完整字段为准，但逻辑删除列统一为 `is_deleted`（与 `application.yml` 及大多数表保持一致）。

- **修改 `data-layer/scripts/mysql/init.sql`**：替换 `exercise` 表定义为 `exercise.sql` 的完整版本，仅将 `deleted` 改为 `is_deleted`。
- **修改 `data-layer/scripts/mysql/mysql-modules/exercise.sql`**：将 `deleted` 列重命名为 `is_deleted`。

> 说明：`application.yml` 已配置 `logic-delete-field: isDeleted`，与 `is_deleted` 列天然映射（`map-underscore-to-camel-case: true`）。若强行改为 `deleted`，需要同时修改 `application.yml` 和另外 9 张表，波及面更大。

#### 2. 修复 `user_exercise` 表名（对应 1.2）

- **修改 `init.sql`**：将表名 `user_exercise` 改为 `user_exercise_record`，与实体 `UserExerciseRecord` 和 Repository SQL 统一。

#### 3. 全局逻辑删除配置与实体脱节（对应 2.1）

**目标**：恢复全局逻辑删除机制，让 `application.yml` 配置真正生效，消除各实体自行处理的混乱。

- **修改 `entity/BaseEntity.java`**：取消注释 `isDeleted` 字段并保留 `@TableLogic`。
- **修改 `entity/Exercise.java`**：删除自有的 `deleted` 字段及 `@TableLogic` 注解，继承 `BaseEntity` 的 `isDeleted` 即可。
- **修改 `service/ExerciseService.java`**：删除所有硬编码的 `deleted` 相关逻辑：
  - `queryWrapper.eq("deleted", 0)` → MyBatis Plus 全局逻辑删除会自动追加，直接删除。
  - `exercise.getDeleted() == 1` → `selectById`/`selectList` 已自动过滤软删除记录，直接删除该判断。
  - `exercise.setDeleted(0)` → 新建实体无需手动设置。
  - `exercise.setDeleted(1)` → 逻辑删除应改为调用 `exerciseRepository.deleteById(exerciseId)`，由 MP 自动处理。
- **修改 `repository/ExerciseRepository.java`**：
  - 自定义 SQL `e.deleted = 0` 改为 `e.is_deleted = 0`。
  - 删除三个未绑定实现的方法（`selectExercisesByKnowledge`、`selectExercisesByUserProgress`、`updateExerciseStatistics`），目前项目中无调用方，属于死代码。如后续需要，再补充 `@Select`/`@Update` 注解或 XML Mapper。

> 说明：启用全局逻辑删除后，MyBatis Plus 会在 `SELECT`/`UPDATE` 中自动追加 `is_deleted = 0`，在 `DELETE` 时自动转为 `UPDATE is_deleted = 1`。Service 层不需要再手写过滤条件，否则是重复工作且容易遗漏。

#### 4. `ExerciseRepository` 自定义 SQL 列名（对应 1.4）

- **修改 `repository/ExerciseRepository.java`**：`findWrongExercisesByUserId` 中的 `e.deleted = 0` 改为 `e.is_deleted = 0`（与统一后的表结构一致）。

### P1 重要（功能缺陷）

#### 5. 编程题判题为空壳（对应 2.2）

**目标**：接入已有的 `CodeExecutionService`，实现真实代码执行与判题。

- **修改 `service/ExerciseService.java`**：注入 `CodeExecutionService`，在 `runTestCases()` 中调用其已有的 `submit()` 方法，传入语言、代码和测试用例。
- **修改 `service/CodeExecutionService.java`**：在 `LANG_MAP` 中补充 `java` 映射（如 `new String[]{"java", "Main.java", "15.0.2"}`），因为种子数据中的编程题均为 Java。

> 说明：`CodeExecutionService.submit()` 已完整实现：调用 Piston 执行、编译错误处理、逐用例比对输出。`ExerciseService` 完全不需要自己再写一套判题逻辑，直接复用即可。

#### 6. 全局异常处理缺失（对应 2.6）

- **新增 `common/GlobalExceptionHandler.java`**（`@ControllerAdvice`）：
  - 捕获 `BusinessException`，返回 `Result.error(code, message)`。
  - 捕获通用 `Exception`，返回 `Result.error("系统异常")`。

> 说明：由于 `ExerciseController` 将统一改为 `Result<T>`（见下一条），全局异常处理只需返回 `Result` 即可覆盖全项目。

#### 7. 响应封装类混用（对应 2.3）

**目标**：全项目统一到 `Result<T>`，消除前端兼容两套格式的负担。

- **修改 `controller/ExerciseController.java`**：将所有方法返回值由 `ApiResponse<T>` 改为 `Result<T>`，方法体内调用 `Result.success(...)`。
- **删除 `common/ApiResponse.java`**（或保留但标记 `@Deprecated`）。

> 说明：`ApiResponse` 比 `Result` 仅多了一个 `timestamp` 字段，其余结构完全一致。由于当前仓库中无前端代码，此改动不会直接影响本仓库构建，但需知会前端同学。

#### 8. 认证硬编码（对应 2.4）

**目标**：在尚未接入 Spring Security / JWT 前，提供最小化过渡方案，避免所有数据挂在固定用户 `1L` 上。

- **修改 `controller/ExerciseController.java`**：在需要用户 ID 的接口方法上增加 `@RequestHeader(value = "X-User-Id", required = false) Long userId` 参数，优先从请求头获取；为空时回退到默认值并打印 `warn` 日志。前端接入登录后只需传 Header 即可过渡，无需大改后端架构。

#### 9. `KnowledgePointController` 直接暴露实体（对应 2.5）

- **新增 `dto/CreateKnowledgePointRequest.java`** DTO，限定可写字段（`name`, `code`, `module`, `level`, `difficulty`, `description`, `corePoints`）。
- **修改 `controller/KnowledgePointController.java`**：`save` 方法接收 DTO 而非 `KnowledgePoint` 实体，配合 `@Valid` 校验（如 `@NotBlank`、`@NotNull`）。

### P2 一般（优化项）

#### 10. 依赖冗余（对应 2.7）

- **修改 `pom.xml`**：移除 `com.alibaba.fastjson2:fastjson2` 依赖。项目实际使用 Jackson，该依赖零引用。

#### 11. MyBatis XML 配置指向空目录（对应 2.8）

- **修改 `application.yml`**：删除 `mapper-locations: classpath*:/mapper/**/*.xml`。`src/main/resources/mapper/` 不存在，保留该配置会在启动时产生不必要的扫描开销。

#### 12. CORS 双重配置且违规（对应 2.9）

- **删除 `config/CorsConfig.java`**：`allowedOriginPatterns("*")` + `allowCredentials(true)` 被浏览器 CORS 规范禁止，会被浏览器拒绝。
- **保留 `config/WebConfig.java`**：限定 `localhost:5173`，符合当前开发场景。

#### 13. `ExerciseService` 手动维护时间戳（对应 2.10）

- **修改 `service/ExerciseService.java`**：删除以下手动赋值：
  - `updateExerciseStatistics` 中的 `exercise.setUpdateTime(LocalDateTime.now())`
  - `saveUserExerciseRecord` 中的 `record.setCreateTime(LocalDateTime.now())`

> 说明：`config/MyMetaObjectHandler` 已配置自动填充，`INSERT` 时自动填 `createTime`，`INSERT`/`UPDATE` 时自动填 `updateTime`。手动赋值不仅多余，还可能覆盖 MP 的填充时机。

#### 14. 密码明文存储（对应 2.11）

- **修改 `controller/UserController.java`**：在 `save()` 中对 `user.getPassword()` 做 BCrypt 哈希（可手动 `new org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder().encode(...)`），或引入 `spring-boot-starter-security`。

> 推荐最小改动：手动使用 `BCryptPasswordEncoder`，避免引入完整 Security 配置链。

#### 15. 实体类继承不一致（对应 2.12）

- **修改 `entity/UserExerciseRecord.java`**：继承 `BaseEntity`，删除自有的 `createTime` 字段。`BaseEntity` 提供 `createTime`、`updateTime` 和统一后的 `isDeleted`。
